Local Services APIs

01

describe your service → create

Describe your service

Timmy types a sentence; AI returns a draft to review — name, classification, suggested service and price. Nothing is created until he confirms; then we commit with the public company-create. We skip the existing one-shot mutation (it commits immediately and spins up a chatbot app a lawn-care business doesn’t need).

coreGenerateBusinessDraft POST /api/v1/business_drafts · NEW build 
# NEW · returns a draft to review — creates nothing
POST /api/v1/business_drafts
{ "prompt": "Lawn mowing for small yards in Manhattan Beach, $25 a mow" }
→ {
    "name": "Lil Timmy's Lawn Mowing", "description": "Small lawns, big care — local kid.",
    "business_type": "services", "industry_group": "home_and_trade_services",
    "suggested_services": [{ "title": "Lawn Mowing", "duration_minutes": 45, "price": 25 }]
  }
# reuses existing AI services: DetermineBusinessModel + GenerateName + GenerateProductMetadata
create business POST /api/v1/companies have · proxy 
# Timmy reviewed/edited the draft → create the business for real
POST /api/v1/companies
{ "title": "Lil Timmy's Lawn Mowing",
  "business_type": "services", "industry_group": "home_and_trade_services" }
→ { "id": "biz_8s2Kd…" }

Not used: the existing internal createCompanyFromPrompt — it’s the one-shot alternative. The draft + create pair replaces it. We call one AI endpoint, not both. The “Service offering” menu choice that precedes this just sets business_type / industry_group defaults — no call.

02

customize offerings

Set name, price & images

Upload the cover art, then create the storefront (a “product”) with pricing inline. Every piece here already ships on /api/v1.

upload image POST /api/v1/files + S3 PUT have 
# 1 · ask for a presigned URL · 2 · PUT the bytes straight to S3
POST /api/v1/files   { "filename": "lil-timmy-poster.jpg", "content_type": "image/jpeg" }
→ { "id": "file_…", "upload_url": "https://s3…" }
PUT {upload_url}   (body: file bytes)
create product POST /api/v1/products have 
POST /api/v1/products
{
  "company_id": "biz_8s2Kd…",
  "title": "Lawn Mowing",
  "route": "lil-timmys-lawn-mowing",
  "gallery_images": [{ "id": "file_…" }],
  "plans": [{ "plan_type": "one_time", "initial_price": "25", "currency": "usd" }]
}
→ { "id": "prod_…", "route": "lil-timmys-lawn-mowing" }
add a tier (optional) POST /api/v1/plans have 
# e.g. a weekly mowing subscription, on top of the one-off above
POST /api/v1/plans
{ "product_id": "prod_…", "plan_type": "recurring",
  "billing_period": 7, "renewal_price": "20", "release_method": "buy_now" }

The rich page is mostly product fields. “What’s Included”, “Why book Timmy”, and the trust badges map to the product’s description, product_highlights & faqs — no new API. Reviews/ratings (the 5.0) are read-only today; collecting them isn’t public. A structured service area (“Manhattan Beach”) has no field yet — a small gap (see Maps/Places).

03

generate product page

Wire the “Book a Mow” + “Message Timmy” CTAs

The page has two CTAs. “Message Timmy” opens a direct line — a public call today. “Book a Mow” needs the booking app: installing it is public, but defining availability and taking a booking live only on the internal app API, so they’re builds.

install booking app POST /api/v1/experiences (+ attach) have 
# 1 · resolve the Calendar Bookings app id
GET /api/v1/apps?…            → { "id": "app_cal…" }
# 2 · create the experience powered by that app, attach it to the product
POST /api/v1/experiences   { "app_id": "app_cal…", "company_id": "biz_…" }   → { "id": "exp_…" }
POST /api/v1/experiences/{exp_id}/attach   { "product_id": "prod_…" }
define availability POST /api/v1/availabilities · NEW build 
# NEW · today this is internal only: /api/apps/calendar_bookings_app/availabilities
POST /api/v1/availabilities?experience_id=exp_…
{ "title": "Lawn Mowing — Book me", "duration_in_minutes": 45,
  "price_per_booking": 25, "timezone": "America/Los_Angeles",
  "timeslots": [{ "day_of_week": "saturday", "starts_time": "09:00", "ends_time": "15:00" }] }
take a booking POST /api/v1/bookings · NEW build 
# NEW · a neighbor books from the page (paid → payment intent, free → instant)
POST /api/v1/bookings   { "availability_id": "avl_…", "starts_at": 1719072000 }
“Message Timmy” POST /api/v1/support_channels (+ messages) have 
# a neighbor opens a direct line to the business
POST /api/v1/support_channels   { "company_id": "biz_…" }   → { "id": "schn_…" }
POST /api/v1/messages   { "channel_id": "schn_…", "content": "Hi! Can you mow this Saturday?" }

“Book a call” = a booking availability with a meeting link — there’s no standalone 1:1 video-booking app. Also, making the booking experience the page’s default CTA (set_primary_experience) has no public endpoint yet — a small extra build.
