First, what counts as “have”

Only /api/v1 is a public API

The public surface is api.whop.com/api/v1/…, produced two ways. If a capability has a path here we have it. If it lives only in internal GraphQL or the /api/apps/* first-party API, a public journey can’t call it — that’s a build.

The journey, call by call

Six steps, in order

01

whop.com → what are you building?

Select “Service offering”

Timmy picks “Service offering” from the list. That choice is a preset — it fills in the classification and which app to install. No call fires yet; these values feed the create call in step 02.

service offering preset no API call have 
# a preset, not a request — these are the real enum values
{
  "business_type": "services",
  "industry_group": "home_and_trade_services",   # lawn care, cleaning, trades
  "apps_to_install": ["calendar_bookings"]      # actually installed in step 04
}

What classification does: business_type / industry_group drive ads account-pooling, discovery search, payment MCC & risk — not the in-product apps. The apps come from installing experiences (step 04). Gap (optional): there’s no first-class “template” model yet.

02

describe your service → create

Describe your service

Timmy types a sentence; AI returns a draft to review — name, classification, suggested service and price. Nothing is created until he confirms; then we commit with the public company-create. We skip the existing one-shot mutation (it commits immediately and spins up a chatbot app a lawn-care business doesn’t need).

coreGenerateBusinessDraft POST /api/v1/business_drafts · NEW build 
# NEW · returns a draft to review — creates nothing
POST /api/v1/business_drafts
{ "prompt": "Lawn mowing for small yards in Manhattan Beach, $40 a mow" }
→ {
    "name": "Lil Timmy's Lawn Mowing", "description": "Small lawns, big care — local kid.",
    "business_type": "services", "industry_group": "home_and_trade_services",
    "suggested_services": [{ "title": "Lawn Mowing", "duration_minutes": 45, "price": 40 }]
  }
# reuses existing AI services: DetermineBusinessModel + GenerateName + GenerateProductMetadata
create business POST /api/v1/companies have · proxy 
# Timmy reviewed/edited the draft → create the business for real
POST /api/v1/companies
{ "title": "Lil Timmy's Lawn Mowing",
  "business_type": "services", "industry_group": "home_and_trade_services" }
→ { "id": "biz_8s2Kd…" }

Not used: the existing internal createCompanyFromPrompt — it’s the one-shot alternative. The draft + create pair replaces it. We call one AI endpoint, not both.

04

generate product page

Add the “Book me / Book a call” CTA

Putting the booking app on the page is a public call today. But the actual booking mechanics — defining availability and taking a booking — live only on the internal app API, so they’re builds.

install booking app POST /api/v1/experiences (+ attach) have 
# 1 · resolve the Calendar Bookings app id
GET /api/v1/apps?…            → { "id": "app_cal…" }
# 2 · create the experience powered by that app, attach it to the product
POST /api/v1/experiences   { "app_id": "app_cal…", "company_id": "biz_…" }   → { "id": "exp_…" }
POST /api/v1/experiences/{exp_id}/attach   { "product_id": "prod_…" }
define availability POST /api/v1/availabilities · NEW build 
# NEW · today this is internal only: /api/apps/calendar_bookings_app/availabilities
POST /api/v1/availabilities?experience_id=exp_…
{ "title": "Lawn Mowing — Book me", "duration_in_minutes": 45,
  "price_per_booking": 40, "timezone": "America/Los_Angeles",
  "timeslots": [{ "day_of_week": "saturday", "starts_time": "09:00", "ends_time": "15:00" }] }
take a booking POST /api/v1/bookings · NEW build 
# NEW · a neighbor books from the page (paid → payment intent, free → instant)
POST /api/v1/bookings   { "availability_id": "avl_…", "starts_at": 1719072000 }

“Book a call” = a booking availability with a meeting link — there’s no standalone 1:1 video-booking app. Also, making the booking experience the page’s default CTA (set_primary_experience) has no public endpoint yet — a small extra build.

Open questions to settle

  • AI draft (B1) — ship as a native /api/v1 endpoint, or as a GraphQL field exposed through the proxy?
  • Booking (B2) — how much of the internal calendar app do we expose publicly: availabilities + bookings only, or the full app surface?
  • Should a neighbor’s booking go through a new /bookings endpoint or be routed through existing checkout?
  • Render vendor — Bannerbear / Placid (programmatic) vs Canva / Adobe Express (editable)? Social — one aggregator (Ayrshare) vs native APIs?
  • Local marketing — Meta/TikTok geo only, or invest in Google Business Profile + Local Services Ads (B6) now?
